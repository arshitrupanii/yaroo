# Backend Project Details

This document explains the backend structure for the Yaroo chat application and documents the new friend search feature.

## Overview

The backend is a Node.js Express API with MongoDB for persistence, JWT cookies for authentication, Cloudinary for image uploads, Socket.IO for real-time messaging, Helmet for security headers, and rate limiting for abuse protection.

Main backend entry point:

```txt
backend/server.js
```

Mounted API groups:

```txt
/api/auth
/api/messages
```

## Backend Folder Structure

```txt
backend/
├── controllers/
│   ├── auth.controller.js       # Signup, login, logout, profile update, auth check
│   └── message.controller.js    # User list, user search, message fetch, message send
├── lib/
│   ├── ApiError.js             # Typed application errors
│   ├── asyncHandler.js         # Async route wrapper for Express 4
│   ├── cloudinary.js            # Cloudinary SDK configuration
│   ├── db.js                    # MongoDB connection helper
│   ├── socket.js                # Express app, HTTP server, Socket.IO setup
│   └── utils.js                 # JWT cookie generation helper
├── middleware/
│   ├── auth.middleware.js       # Reads JWT cookie and attaches req.user
│   ├── error.middleware.js      # Global 404 and error response handling
│   └── requestId.middleware.js  # Adds x-request-id to every request/response
├── model/
│   ├── message.model.js         # Message schema
│   └── user.model.js            # User schema
├── routes/
│   ├── auth.routes.js           # Auth route definitions
│   └── message.routes.js        # Message and user route definitions
└── server.js                    # App middleware, route mounting, DB connect, server start
```

## Server Startup Flow

`backend/server.js` does the following:

1. Loads environment variables using `dotenv`.
2. Configures CORS in non-production mode using `FRONTEND_URL`.
3. Adds JSON, URL-encoded body parsing, and cookie parsing middleware.
4. Mounts auth routes at `/api/auth`.
5. Mounts message routes at `/api/messages`.
6. Adds a `/health` endpoint.
7. Handles API 404 errors.
8. Serves `frontend/dist` in production.
9. Handles global errors in one middleware.
10. Connects to MongoDB.
11. Starts the HTTP and Socket.IO server.

## Environment Variables

The backend expects these variables:

```txt
MONGODB_URI=mongodb://localhost:27017/yaroo
PORT=5000
JWT_SECRET=your-secret
FRONTEND_URL=http://localhost:5173
PASSWORD_RESET_URL=http://localhost:5173
NODE_ENV=development
CLOUDINARY_NAME=your-cloudinary-name
CLOUDINARY_API_KEY=your-cloudinary-key
CLOUDINARY_API_SECRET=your-cloudinary-secret
```

Note: `.env.example` uses `CLOUDINARY_NAME`, and `backend/lib/cloudinary.js` reads `CLOUDINARY_NAME`.

In production, `JWT_SECRET` must be at least 32 characters.

## Authentication Flow

Authentication is cookie-based.

1. User signs up or logs in.
2. `generateToken` creates a JWT with `{ userId }`.
3. The JWT is stored in an HTTP-only cookie named `ChatAppToken`.
4. The JWT includes issuer and audience checks.
5. Protected routes use `authMiddleware`.
6. `authMiddleware` verifies the cookie, fetches the user, removes sensitive fields, checks whether the password changed after token issue, and assigns the user to `req.user`.

Protected routes require the `ChatAppToken` cookie.

Security hardening currently includes:

```txt
Helmet security headers
General /api rate limit
Stricter auth route rate limit
Stricter forgot/reset password rate limit
Password hidden by default in User queries
Generic login failure message
Generic forgot-password response to reduce account enumeration
Hashed password-reset token storage
Production JWT secret length check
Global error handling with request IDs
Async route wrapper for rejected promises
Central Mongoose validation, cast, duplicate key, payload, and JSON parse error mapping
```

## Global Error Management

Global error handling is implemented with these files:

```txt
backend/lib/ApiError.js
backend/lib/asyncHandler.js
backend/middleware/error.middleware.js
backend/middleware/requestId.middleware.js
```

Every request gets an `x-request-id` response header. The same id is included in error JSON so frontend errors can be matched with backend logs.

Standard error response shape:

```json
{
  "message": "Invalid email format",
  "code": "VALIDATION_ERROR",
  "requestId": "uuid"
}
```

Some errors can also include field-level details:

```json
{
  "message": "Validation failed",
  "code": "VALIDATION_ERROR",
  "requestId": "uuid",
  "details": [
    {
      "field": "email",
      "message": "email already exists"
    }
  ]
}
```

In development, errors include a stack trace. In production, stack traces are hidden from API responses.

Controllers should throw `ApiError` for expected user-facing failures:

```js
throw new ApiError(400, 'Username is required', { code: 'VALIDATION_ERROR' });
```

Route handlers should be wrapped with `asyncHandler` so async errors reach the global error middleware:

```js
router.post('/login', asyncHandler(login));
```

## Auth API

Base path:

```txt
/api/auth
```

### POST `/signup`

Creates a new user.

Request body:

```json
{
  "firstname": "Arshit",
  "email": "arshit@example.com",
  "password": "Secret123"
}
```

Response includes:

```json
{
  "_id": "user-id",
  "firstname": "Arshit",
  "email": "arshit@example.com",
  "profilePicture": ""
}
```

### POST `/login`

Logs in an existing user.

Request body:

```json
{
  "email": "arshit@example.com",
  "password": "Secret123"
}
```

### POST `/logout`

Clears the `ChatAppToken` cookie.

### POST `/forgot-password`

Creates a short-lived password reset token.

Request body:

```json
{
  "email": "arshit@example.com"
}
```

Response is generic so attackers cannot confirm whether an email exists:

```json
{
  "message": "If an account exists for that email, a password reset link has been created"
}
```

In non-production mode only, the API also returns `resetToken` and `resetUrl` so the feature can be tested without an email provider.

### POST `/reset-password/:token`

Resets the user's password with a valid, unexpired token.

Request body:

```json
{
  "password": "NewSecret123"
}
```

Behavior:

1. Hashes the incoming token.
2. Finds a user with a matching stored token that has not expired.
3. Hashes the new password.
4. Clears reset token fields.
5. Sets `passwordChangedAt`.
6. Issues a fresh auth cookie.

### PUT `/update-profile`

Protected route. Updates the current user's profile picture by uploading the image to Cloudinary.

Request body:

```json
{
  "profilePicture": "base64-image-data"
}
```

### GET `/check`

Protected route. Returns the authenticated user from `req.user`.

## Message And User API

Base path:

```txt
/api/messages
```

All routes in this group are protected by `authMiddleware`.

### GET `/users`

Returns all users except the logged-in user.

Used by the sidebar user list.

### GET `/search/:username`

Searches users by name for the friend search feature.

Example:

```txt
GET /api/messages/search/ar
```

Behavior:

1. Reads `username` from the URL.
2. Trims the search text.
3. Searches `User.firstname` case-insensitively.
4. Excludes the logged-in user.
5. Removes `password`, `createdAt`, and `updatedAt` from results.
6. Limits results to 10 users.

Example response:

```json
[
  {
    "_id": "user-id",
    "firstname": "Arshit",
    "email": "arshit@example.com",
    "profilePicture": ""
  }
]
```

Current implementation file:

```txt
backend/controllers/message.controller.js
```

Current route file:

```txt
backend/routes/message.routes.js
```

Important route order:

```txt
GET /users
GET /search/:username
GET /:id
POST /send/:id
```

`/search/:username` must stay before `/:id`, because `/:id` is a dynamic route and can catch other paths if placed too early.

### GET `/:id`

Returns messages between the logged-in user and another user.

Example:

```txt
GET /api/messages/64f...
```

### POST `/send/:id`

Sends a message to another user.

Request body:

```json
{
  "text": "Hello",
  "image": "optional-base64-image-data"
}
```

If `image` is present, it is uploaded to Cloudinary and the saved message stores the Cloudinary URL.

After saving a message, the backend checks if the receiver is online and emits `newMessage` through Socket.IO.

## Socket.IO Flow

File:

```txt
backend/lib/socket.js
```

Socket behavior:

1. Frontend connects with `userId` in `socket.handshake.query`.
2. Backend stores `userId -> socket.id` in `userSocketMap`.
3. Backend emits `getOnlineUsers` with all online user ids.
4. On disconnect, the user is removed from `userSocketMap`.
5. When a message is sent, the backend emits `newMessage` to the receiver if they are online.

## Friend Search Feature

The backend now supports searching friends by username-like text using the existing `firstname` field.

Endpoint:

```txt
GET /api/messages/search/:username
```

Frontend usage example:

```js
const res = await Axiosinstance.get(`/messages/search/${searchText}`);
```

Recommended frontend behavior:

1. Add a search input in the sidebar.
2. Debounce requests by 300ms.
3. If the input is empty, call `/messages/users` to show the normal sidebar list.
4. If the input has text, call `/messages/search/:username`.
5. Store the response in the same `users` state used by the sidebar list.

## Notes For Future Improvements

The current `User` model does not have a separate `username` field. If the app needs real unique usernames later, add a `username` field to `user.model.js`, validate it during signup, create a unique index, and update the search query to search `username` instead of `firstname`.

For larger user collections, prefer indexed search fields instead of an unanchored regular expression query.
